## 一、项目定位

新版 Sim-U-Kraft 不是对旧版代码的逐文件搬运，而是以旧版玩法为需求来源，用新的架构重新实现一个更稳定、更易扩展、更高性能的城市模拟模组。

核心定位：

- 保留旧版玩家体验：城市核心、建筑盒、NPC、市民生活、建筑施工、经济与生产链。
- 重写底层架构：服务层、SQLite 持久化、网络同步、客户端缓存、LDLib 界面分层。
- 服务端永远是权威方：资金、领地、雇佣、建筑、NPC、物品变化都必须服务端验证。
- 客户端只做显示与交互请求：HUD、地图、建筑预览、GUI、缓存都不能作为最终数据来源。
- 长期数据统一进入 SQLite：SavedData 只作为生命周期桥接和运行期入口，不再承担复杂业务规则。

当前阶段可以概括为：

```text
基础框架已经可用，城市、区块、地图、NPC、建筑盒、住宅控制盒与 SQLite 链路已经打通。
项目重点正在从“底层能跑”转向“玩法闭环可玩”和“长期可维护”。
```

## 二、已经稳定的设计原则

### 1. 服务层优先

GUI、事件、网络包只负责收集输入和发送请求，业务逻辑进入 Service。

当前已经形成的服务入口包括：

| 模块 | 主要类 |
|---|---|
| 城市 | `CityService`、`CityManager` |
| 区块领地 | `CityClaimService`、`CityChunkManager`、`CityChunkSyncService` |
| 经济流水 | `EconomyService`、`FinanceLedgerService` |
| 城市 POI | `CityPoiService`、`CityPoiManager` |
| NPC | `CitizenService`、`CitizenManager`、`CitizenHousingService` |
| NPC 休息/传送 | `CitizenHomeRestService`、`CitizenTeleportService` |
| NPC 死亡 | `CitizenDeathService` |
| 职业/雇佣 | `CityJobAssignmentService`、`CityJobCapacityService`、`CityJobMobilityService` |
| 建筑 | `BuildingStructureService`、`BuilderConstructionService`、`PlacedBuildingService` |
| 住宅 | `ResidentialControlBoxService`、`ResidentialBedPoiService` |
| HUD 同步 | `HudSyncService` |

### 2. SQLite 作为长期存储

当前长期数据以 SQLite 为主，模块化仓库负责具体读写。

当前主要仓库：

```text
CitySqliteRepository
CityChunkSqliteRepository
CityPoiSqliteRepository
CitizenSqliteRepository
BuildingTaskSqliteRepository
BuildingStructureRepository
```

当前主要数据：

```text
cities
city_members
finance_transactions
city_chunks
city_pois
citizens
citizen_skills
building_tasks
building_structures
placed_buildings
```

后续新增长期系统时，应优先新增关系表和索引，不再新增散落 JSON、临时 NBT 或全局静态缓存作为主存储。

### 3. 缓存必须有作用域

除建筑资源缓存 `simukraftbuilding` 外，其他缓存不能跨存档通用。

必须按需要带上：

```text
saveId
dimensionId
cityId
playerId
```

客户端缓存只用于显示，不参与最终判定。

### 4. 旧版优先作为规则参考

旧版规则能查到时，优先还原旧版玩家体验；如果旧版实现存在明显性能问题，只保留玩法规则，不继承技术实现。

禁止直接继承的典型旧问题：

- GUI 内直接写业务逻辑。
- 网络包直接改存档。
- 全局静态缓存跨世界串数据。
- 每 tick 全量扫描 NPC、建筑、仓库或世界方块。
- 大型管理类承担注册、网络、存档、UI 多重职责。

## 六、开发注意事项

### 1. 服务端安全

所有会改变世界或数据的操作都必须验证：

- 玩家是否存在。
- 维度是否正确。
- 方块是否仍存在。
- 距离是否合理。
- 城市权限是否足够。
- 城市资金是否足够。
- 目标状态是否仍有效。

客户端传来的数据只能作为请求参数，不能直接信任。

### 2. 性能要求

必须避免：

- 每 tick 全量扫描所有 NPC。
- 每 tick 扫描建筑区域。
- 每帧重建大型预览网格。
- 地图每帧大规模 CPU 绘制。
- 大列表 GUI 每次操作都全量请求服务端。
- 没有作用域的全局静态缓存。

推荐方式：

- 使用索引缓存。
- 使用 dirty 标记。
- 使用分片 tick。
- 使用增量保存。
- 使用按需同步。
- 使用服务层集中失效缓存。

### 3. 数据一致性

必须保证：

- SQLite 是长期事实来源。
- Manager 内存数据启动时从 SQLite 恢复。
- 关键变更立即增量写入。
- 定时全量同步只做兜底。
- 服务停止时做最终保存。
- 客户端缓存掉线或切世界时清理。

### 4. GUI 规范

GUI 只做：

- 显示服务端给出的数据。
- 发起请求。
- 做客户端视觉反馈。

GUI 不做：

- 扣钱。
- 改城市归属。
- 改 NPC 职业。
- 直接写 SQLite。
- 直接修改 Manager 内部集合。

### 5. 旧版兼容原则

允许复用：

- 建筑模板。
- 资源文件。
- 语言 key 思路。
- 可见交互流程。
- 性能较好的独立算法思路。

不建议复用：

- 巨型 GUI 类。
- 混合业务逻辑的工具类。
- 直接写存档的 packet。
- 多套重复雇佣数据。
- 无作用域缓存。

## 八、性能验收目标

| 场景 | 目标 |
|---|---|
